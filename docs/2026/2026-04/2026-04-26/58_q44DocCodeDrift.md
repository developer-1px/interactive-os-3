# Q44 — CANONICAL.md와 코드 drift 막는 법?

## 질문

문서가 코드를 안 따라잡으면 문서는 거짓말이 된다. CANONICAL.md가 코드와 어긋나는 drift를 어떻게 막는가? 문서 우선인가 코드 우선인가?

## 해보니까 알게 된 것

- ds의 답: **문서가 코드를 묶고, 코드가 문서를 검증한다**. 단방향이 아니라 양방향 직렬화.
- 코드 → 문서: `/canonical` 스킬이 코드를 훑어 일탈을 보고하고, 정본 갱신 시 CANONICAL.md 1행을 갱신하며 변경 이력에 일자 기록. 코드 변경이 문서 갱신을 강제한다.
- 문서 → 코드: 정본 1행마다 lint가 1:1로 붙어 있어(Q43) 문서에 적힌 형태가 lint 통과의 정의가 된다. 문서가 못 따라잡으면 lint가 못 짜진다 → 머지 안 됨.
- 핵심 장치 3개:
  1. **CANONICAL.md 1행 = 1 lint** 메타-규칙 (지향). 문서 행이 검증 가능한 단언 형태("X만 허용", "Y 금지")로 적힘.
  2. **변경 이력 일자 박제** — drift가 생기면 마지막 일자가 오래된 정본 행이 의심 대상.
  3. **`/canonical` 감사 + handoff A0 역PRD** — 머지 직전 문서 ⊃ 구현 검증.
- 그래도 drift는 생긴다. drift가 발견되면 정본을 갱신할지(코드가 옳음), 코드를 수렴할지(문서가 옳음) 둘 중 하나로 끊는다 — 양립을 허용하지 않음.
- "문서 우선이냐 코드 우선이냐"는 잘못된 질문. 정답은 "**다른 쪽으로 옮길 수 없는 사실은 정본이 아니다**"다. 직렬화 원칙이 여기서도 작동한다.

## 근거

- CANONICAL.md L106-112 "변경 이력" — 일자별 정본 갱신이 코드 일탈 수렴과 함께 기록됨
- CANONICAL.md L92-97 갱신 절차 4단계 마지막이 "일탈 감사 재실행"
- `/canonical` 스킬 설명: "정본을 미리 선언하고 거기서 벗어난 모든 사례를 끌어당기기"
- `/handoff` A0 역PRD: Blueprint ⊃ Implementation 검증 — 문서가 구현을 포함하는지 강제

## 남은 의문

- CANONICAL.md 행에 lint 파일 경로를 명시하는 메타-필드가 필요한가 (문서-lint 1:1 추적)
- 문서 갱신 PR 없이 코드만 갱신되는 PR을 hook으로 차단할지
- 문서 행 형식을 더 기계 가독적으로(`yaml`/`toml`) 바꿀지 — 지금은 마크다운 자유 문장
