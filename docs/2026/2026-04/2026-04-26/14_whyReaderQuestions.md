# Why 시리즈 — 독자가 당연히 품을 질문 목록

## 배경

`08_projectWhy` ~ `13_whyDeclarativeSerialization`까지의 6개 문서를 처음 접한 외부 독자(시니어 프론트엔드 엔지니어, 디자인 시스템 관리자, LLM 도구 빌더, 디자이너, 백엔드 출신, 신입)가 자연스럽게 떠올릴 질문을 망라한다. 각 질문은 후속 why 문서·반례 토론·정본 갱신의 입력이 된다. **여기서는 답하지 않는다.** 답하기 시작하면 또 다른 문서가 된다.

## 내용

### A. 전제 의심 — "정말 그래?"

1. LLM이 코드를 잘 못 짠다는 게 전제인데, 모델이 더 좋아지면 ds의 why가 무너지지 않나?
2. "동일 의도 → 동일 출력 분포"는 측정 가능한가? 어떻게 검증하나?
3. 학습 분포가 편향됐다고 단언하는 근거는? Anthropic·OpenAI 모델이 같은 분포를 학습했다고 어떻게 아나?
4. "선택지가 줄면 LLM 결정성이 올라간다"는 가설은 실제 실험으로 입증됐나, 아니면 직관인가?
5. 사람을 위한 자유도와 LLM을 위한 결정성이 정말 trade-off인가? 둘 다 잡는 길은 없나?
6. 디자인 시스템이 풀 문제가 LLM 결정성이라는 framing 자체가 over-fitting 아닌가?
7. 5년 후에도 이 why가 유효할까? — 모델 컨텍스트가 무한이면 "어휘 일관성"이 무의미해질 수도

### B. 비교 — "기존 시스템은 왜 안 되는데?"

8. shadcn/ui는 copy-paste 기반인데 ds와 무엇이 다른가?
9. Radix를 그대로 쓰면 안 되나? 왜 그 위에 또 한 층을 얹나?
10. Material 3·Polaris·Carbon은 이미 큰 시스템인데 ds가 이긴다고 주장하는 점은?
11. Tailwind + shadcn 조합이 사실상 LLM이 가장 잘 짜는 스택인데, ds는 그것보다 결정성이 높다는 증거가 있나?
12. Storybook·Figma Token이 풀려는 문제와 ds가 풀려는 문제는 같은가 다른가?
13. Web Components·Lit·Stencil 같은 표준 진영과의 관계는?
14. shadcn처럼 사용자가 컴포넌트 코드를 소유하는 모델이 ds와 충돌하나?

### C. 실제 적용 — "이걸로 진짜 앱을 만들 수 있나?"

15. variant 0개·1 role = 1 component이면 컴포넌트 수가 폭발하지 않나? 몇 개까지 늘어나나?
16. design 마이너 변경(예: 마케팅 페이지 hero CTA만 살짝 다르게)을 어떻게 처리하나?
17. A/B 테스트로 두 시각 변형이 필요할 때는?
18. dark mode·국제화·고대비 접근성 같은 cross-cutting concerns는 어떻게?
19. 화면 전체가 제품 특수 도메인일 때 (예: 캘린더·차트·코드 에디터) 정본이 어디까지 커버하나?
20. 디자이너가 Figma에서 ds 외 형태를 가져오면 어떻게 처리하나?
21. SaaS multi-tenant에서 테넌트별 시각 customization은 어떻게?
22. 마이그레이션 — 기존 MUI 앱을 ds로 옮길 때 점진 전환 경로는?
23. 서드파티 컴포넌트(react-select, react-table)와 어떻게 공존하나?
24. 디자인 토큰을 외부(Figma·디자인 도구)와 어떻게 동기화하나?
25. SSR·streaming·React Server Component 환경에서 정본은 그대로인가?

### D. 트레이드오프 — "포기한 것의 대가는?"

26. variant 금지 → 이름 폭증. 어휘가 커지면 LLM이 이름 자체를 잘못 부르지 않나?
27. children 금지 → data 트리 폭증. 큰 데이터 트리를 매번 만드는 메모리·런타임 오버헤드는?
28. de facto 표준 채택 → ds가 후행. 새로운 패턴이 나오기 전까지 ds도 못 채택한다는 한계는?
29. 직렬화 가능 → 함수 prop 추방. 진짜로 함수가 필요한 곳(custom render)은 어떻게 우회하나?
30. classless → CSS-in-JS·Tailwind 생태계와 단절. 학습 자료·도구가 빈약해지지 않나?
31. escape hatch 0개 → "잠깐만 빨리" 코드가 불가능. 프로토타이핑 속도는 떨어지지 않나?
32. 1 role = 1 component → 같은 의미·다른 시각이 진짜 필요할 때 새 role을 발명해야 한다. 발명 비용은?
33. ds 자체의 학습 곡선이 가파른 거 아닌가? 신입 온보딩은?

### E. 회색 지대 — "여기는 어떻게 결정?"

34. 두 라이브러리가 갈리는 이름(예: `Trigger` vs `Activator`)은 어떻게 결정?
35. de facto 표준이 ds 원칙과 충돌할 때(예: variant를 모두가 쓰는 패턴) 어디로 가나?
36. 새 ARIA pattern이 나왔는데 아직 라이브러리 수렴이 없을 때는 채택? 보류?
37. 같은 의미인데 layer가 다른 경우(예: form Submit vs dialog Confirm) — 같은 컴포넌트? 다른 컴포넌트?
38. "사용처 1곳도 entity 승격"이 만든 entity 묘비가 정본 어휘를 오염시키지 않나?
39. CANONICAL.md C6 ("정본 ≠ 이상형")과 P1 ("variant 금지") 사이에서 — 코드에 한 곳도 안 쓰이는 정본 컴포넌트는 죽이나, 두나?
40. 정본이 갈등할 때(예: 직렬화 vs de facto 표준) 우선순위는?

### F. 메타 — "이 시스템 자체에 대한 질문"

41. 정본 갱신 누가 결정? 1인 합의? 팀 합의? 어떤 절차?
42. "임시"·"유산"으로 분류된 형태는 언제 정리? 영원히 묻혀있는 건 아닌가?
43. 정본 위반을 자동으로 잡는 lint·hook은 얼마나 커버하나? 사람이 PR에서 잡는 비율은?
44. 정본 문서(CANONICAL.md)와 코드의 drift를 어떻게 막나?
45. ds 자체가 외부에 공개되면 (오픈소스) 외부 contributor가 정본을 흔들 때 정책은?
46. ds의 성공 지표는 뭔가? "LLM 생성 코드가 사람 코드와 같다"를 어떻게 측정?
47. 이 why 시리즈 자체가 정본인가? 시간이 지나면 갱신되나?
48. why 시리즈를 읽고 ds를 받아들이지 않은 사람과 협업할 때 어떻게?

### G. 철학·사상 — "이건 이념인가 엔지니어링인가?"

49. "Decision-shrinking system"은 결국 권력의 중앙집중 아닌가? 누가 정본을 결정하는지가 곧 권력이다
50. LLM에 적응하는 시스템 = LLM이 제품을 만드는 미래를 받아들이는 것. 이게 디자이너·개발자의 직업적 함의는?
51. "사람의 자유도를 줄여 LLM 결정성을 높인다"는 인간을 LLM의 보조로 두는 framing 아닌가?
52. ds의 why는 의도적으로 "왜 이렇게 만드는가"만 답한다. "왜 이런 제품을 만드는가"는 어디에?
53. 모든 정본이 결국 "통제 가능성"을 향한다. 통제 불가능한 표현(예술적 표현·개인 취향)은 ds 안에서 죽나?
54. 데이터로 환원되지 않는 디자인 가치(분위기·brand feel)는 어디로 가나?

### H. 작은 의문 — "사소하지만 신경 쓰이는"

55. `data-part`가 결국 className의 다른 이름 아닌가? 무엇이 본질적으로 다른가?
56. ARIA로 모든 상태를 표현한다고 하지만 ARIA가 부족한 경우(예: drag preview)는 어떻게?
57. `useResource`의 dispatch 함수는 직렬화 불가인데, C2 위반 아닌가?
58. `definePage` entity tree는 결국 JSX의 이름만 바꾼 것 아닌가?
59. `defineFlow` 한 줄이 진짜 한 줄로 끝나나? 실제 복잡한 wiring 사례는?
60. de facto 4개 라이브러리만 참조 — Headless UI·Reach UI·Aria Components 등은 왜 빠짐?
61. why 시리즈 6개의 일관성 — 서로 모순되는 부분 없나? (예: "발명하지 않는다" vs "ds 고유 어휘 도입")
62. Korean·English 어휘 혼용 — 정본은 어느 언어로?

- 이 목록 자체를 정본 갱신의 압력 측정기로 활용 (반복 질문 = 정본 부족)
