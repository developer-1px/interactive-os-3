# Q51 — 사람을 LLM 보조로 두는 framing 아닌가

## 질문

"사람의 자유도를 줄여서 LLM의 결정성을 높인다." 이 문장은 주어가 LLM이고 사람이 보조다. 인간 중심이 아니라 LLM 중심 framing 아닌가?

## 해보니까 알게 된 것

문장만 보면 그렇게 읽힌다. 그러나 무엇을 위해 LLM 결정성을 높이는지를 따라가면 끝점은 사람에게 돌아온다.

- LLM 결정성이 올라가면 → 같은 의도가 같은 출력으로 가고
- 같은 출력으로 가면 → 사람이 결과를 검증·예측·신뢰할 수 있고
- 신뢰할 수 있으면 → 사람이 "노예로 따라잡는 검토자"가 아니라 **결정자**로 일할 수 있다

반대 framing을 보자. "사람의 자유도를 최대한 보장한다 → LLM은 매번 다른 출력을 낸다 → 사람이 매번 후처리한다 → 결국 사람이 LLM 출력의 청소부가 된다." 이게 진짜 "사람이 LLM 보조"다.

내 입장: ds의 framing은 "사람을 LLM의 보조로 두는 것"의 **반대**다. **LLM을 사람이 통제 가능한 도구로 만드는 것**이다. 자유도를 양보하는 대상이 사람이 아니라 코드의 표현 공간이라는 점이 핵심이다 — 사람의 의사결정 자유도는 오히려 늘어난다 (어떤 것이 정본이어야 하는가).

다만 한 가지는 인정한다. **공짜는 아니다**. ds 어휘를 학습하는 비용, "왜 이 형태인지" 익히는 비용은 사람이 낸다. 그 비용을 내고 통제권을 사는 거래다. 비용을 안 내고 통제권을 갖고 싶다면 ds는 맞지 않다.

## 근거

- C1~C6 헌장 — 모든 항목이 "결과의 예측 가능성"을 사람이 갖기 위한 제약. LLM 편의가 아니라 사람의 검증 가능성이 목적
- ControlProps(data, onEvent) — 사람이 데이터로 UI를 기술하면 LLM이 옮긴다. 결정은 사람, 옮김은 LLM
- "minimize choices for LLM" 메모리 — 선택지를 줄이는 것은 LLM의 자유를 빼앗는 것이지 사람 자유를 빼앗는 게 아니다
- defineFlow·useFlow 한 줄 — 사람이 wiring을 한 줄로 결정하면 LLM이 부풀린다. 결정 권한은 사람에게

## 남은 의문

- "결정만 하는 사람"이 만족스러운 직무 경험인가 — "만들기의 즐거움"을 잃는다는 반론
- LLM이 결정자(canonicalist)까지 잘 하게 되면 framing이 다시 흔들린다 — Q49로 회귀
- ds 어휘 학습 비용을 누가 부담하는지에 따라 이게 인간 중심인지 자본 중심인지 갈린다 (사용자 vs 회사)
