# 자가발전 수렴 루프 레이어의 컨셉맵

## 배경

ds 프로젝트는 LLM이 코드를 짤 때 *같은 실수를 다음 턴에 구조적으로 못 하도록* 하는 자가발전 수렴 루프를 목표로 한다. 현재 5종 lint + Stop 훅 시각 아카이브 + 50+ skill로 닫힌 ratchet의 절반은 만들어져 있으나, **양의 규칙(=요구 invariant)** 을 두 번 시도해 두 번 실패했다. 진단: 규약을 1개의 통로로 만들지 않은 상태에서 양의 규칙을 걸면 해치(escape hatch)가 범람하여 루프 자체가 망가진다.

이 문서는 그 레이어의 **풀어야 할 문제 지도**를 정리한다. 해법이 아니라 *문제의 구조*를 명시하는 것이 목적.

## 내용

### 1. 레이어 정의 — 무엇을 만들고 있는가

> "통로 단일화로 양의 규칙을 가능하게 하는 메타 레이어"

자가발전 수렴 루프의 본질은 **음의 규칙(금지)** 만으로는 닫히지 않는다. 음의 규칙은 통로 수에 무관하게 grep 한 번으로 닫히지만, *행위*가 맞는지(=양의 규칙)는 행위가 들어오는 통로마다 다르게 검증해야 한다. 따라서 자가발전 루프의 다음 단계는:

```
(a) 통로 후보 생성 → (b) 우회로 마이그레이션 → (c) 측정 가능화 → (d) 통로의 마찰 < 우회로의 마찰 → (e) 양의 규칙 점화
```

이 5단계 파이프라인이 **차원별로 어디까지 진행됐는지**를 통합 관리하는 레이어가 지금 미정의 상태다.

### 2. 차원 인벤토리 — 현재 상태 실측

| 차원 | 통로 후보 | 우회로 잔존 | 단계 |
|---|---|---|---|
| role 발급 | `ui/*` | routes raw role 7건(문서) | ✅ 수렴 |
| state | `style/states/selectors.ts` | 0 | ✅ 수렴 |
| token | `fn/*` + `var(--ds-*)` | 0 | ✅ 수렴 |
| layout | `definePage` | Row/Column JSX 잔존 | ⚠️ 수렴 중 |
| **kbd** | `composeAxes`/`useRoving` | `onKeyDown` 14파일 | ❌ 통로 2개 공존 |
| **data** | `ControlProps` | `children` 잔존 | ❌ 통로 2개 공존 |
| **focus** | — | 각자 알아서 | ❌ 통로 후보 부재 |

수렴한 3차원과 수렴 못 한 3차원의 차이는 lint 강도가 아니라 **통로의 자연 마찰**에 있다.

### 3. 미해결 문제 — 컨셉맵

#### P1. (d) 마찰 역전 문제 [최상위]

> 통로가 우회로보다 *작성하기 쉬워야* 한다. 그렇지 않으면 음의 lint를 백 개 걸어도 해치가 자란다.

- kbd: `onKeyDown`이 React 기본값. `composeAxes`는 import + 설정 필요 → 마찰 역전.
- data: `children`이 React 기본값. `ControlProps`는 데이터 정규화 필요 → 마찰 역전.
- 풀어야 할 것: **통로를 ui 컴포넌트의 root에 자동 attach**하여 소비자가 import조차 안 해도 default가 통로가 되게.

#### P2. "통로 2개 공존" 국면 처리

> 통로 0개(우회 명시) < 통로 1개(강제) << 통로 2개(조용히 양립)
> 가장 나쁜 국면이 *지금 ds의 kbd/data*다.

- 두 통로가 공존하면 lint가 false positive를 토하기 시작 → 양의 규칙이 *해치 펌프*가 된다.
- 풀어야 할 것: **2개 공존 차원을 0개 또는 1개로 강제 환원**하는 마이그레이션 레인. "통로를 추가한다"보다 "둘 중 하나를 죽인다"가 우선.

#### P3. 측정 비대칭 문제

> 차원이 *측정 비대상*이면 통로 단일화가 미정의 문제가 된다.

- role: `grep 'role="'` → 음수 측정 가능
- kbd: `onKeyDown`만 grep해선 부족 (`addEventListener`, hotkey lib, `keydown` 캡처 등 다중 진입점)
- focus: 통로 후보조차 없어 측정 자체 불가
- 풀어야 할 것: **차원별 잔존 계량기 정의**. 측정 가능하지 않은 차원은 양의 규칙 논의 자체를 보류.

#### P4. 양의 규칙 추가 시점의 게이트 부재

> 두 번의 실패는 (a)~(d) 미통과 상태에서 (e)부터 시도한 결과로 추정된다.

- 풀어야 할 것: **lint 추가 전 체크리스트**. "이 차원의 통로 수가 1인가? 잔존 계량기가 0인가? N턴 동안 0 유지했는가?" 셋 다 yes일 때만 양의 규칙 추가.

#### P5. 수렴 oracle의 부재

> "수렴됐다"의 정량 정의가 없어 사람이 눈으로 판정한다 → reward signal이 외부에 있음 → "자가발전" 미달.

- timeline.jsonl은 변경 hash만 기록.
- 풀어야 할 것: **차원별 잔존 카운트 시계열**을 timeline에 합성. "kbd 차원 onKeyDown 잔존 14 → 0 N턴 유지" 같은 곡선이 보여야 한다.

#### P6. 메타 통로 자체의 단일화

> ds 차원은 단일 통로를 강제하면서 *메타 시스템(skill 50+, /go /auto /handoff /improve 등)* 은 통로가 N개. 본인 원칙과 충돌.

- 풀어야 할 것: 메타 레이어에도 같은 (a)~(e)를 적용. 어느 skill이 invocation되지만 수렴 기여 0인지 측정 → 통로 정리.

#### P7. 메모리·skill·hook의 부패 가드

> 메모리 entry, skill 정의, hook 규칙은 시간이 가면 stale.

- 풀어야 할 것: TTL 또는 자동 검증 루프. 잔존 계량기가 0이 된 차원의 음의 lint는 *자동으로 비활성화/승격*되어야 한다(=잠금에서 invariant로).

### 4. 컨셉맵 (의존 관계)

```
P3(측정) ──┐
           ├──→ P1(마찰 역전) ──→ P2(2개 공존 해소) ──→ P4(게이트) ──→ P5(oracle)
P6(메타) ──┘                                                              │
                                                                          ▼
                                                                       P7(부패 가드)
```

- P3는 모든 후속의 전제 — 측정 못 하면 아무것도 못 한다.
- P1은 P2의 전제 — 마찰을 안 뒤집으면 우회로는 안 죽는다.
- P4는 P1·P2·P3가 함께 만족된 차원에서만 발동.
- P5는 P4가 차원별로 누적되어야 시계열로 보임.
- P7은 P5의 신호를 받아 lint를 자동 진화시키는 메타루프.

### 5. 핵심 통찰 (압축)

1. **음의 규칙은 차원 수에 무관, 양의 규칙은 통로 수에 비례해 부패한다.**
2. **마찰 역전이 풀리지 않은 차원에 lint를 더하면 해치 펌프가 된다.**
3. **통로 0개보다 2개가 더 위험하다.** 첫 일은 "추가"가 아니라 "둘 중 하나를 죽이기".
4. **측정 가능성은 수렴 가능성의 전제**다. 측정 비대상 차원은 양의 규칙 논의 자체를 보류.
5. **수렴 oracle이 외부(=사람)에 있는 한 자가발전이 아니다.** 잔존 계량기를 timeline에 합성해야 reward signal이 내재화된다.
