# When NOT to use ds

ds는 모든 팀·모든 프로젝트에 맞지 않는다. 다음 상황에서는 다른 도구가 더 낫다.

## 1. 페이지마다 시각이 달라야 하는 마케팅·아트 디렉션 사이트

랜딩 페이지·캠페인 페이지·브랜드 사이트처럼 **페이지마다 의도적으로 다른 모습**이 필요한 제품에서 ds는 방해가 된다. ds는 "1 role = 1 component, 같은 역할은 같은 모습"이 정본이다. 시각적 다양성을 자유롭게 조립해야 한다면 Tailwind + shadcn·바닐라 CSS·CMS 기반 구성이 더 적합하다.

## 2. TypeScript·lint 문화가 정착되지 않은 팀

ds의 강제력은 TypeScript의 타입 추론 + lint 규칙 + audit 스크립트로 작동한다. `// @ts-ignore`·`/* eslint-disable */`이 일상화된 코드베이스에 ds를 얹으면 정본이 무너진다. 정본 = 코드가 통과해야만 머지되는 상태다. lint를 우회하는 팀에서는 ds가 약속을 지킬 수 없다.

## 3. 극단적 페이지별 시각 변형이 제품의 본질인 경우

캔버스·디자인 툴·노코드 빌더처럼 사용자가 임의 시각을 만드는 제품은 ds의 정본 외부에 있는 자유도가 본질이다. 이 경우 ds는 그 제품의 **빌더 UI**(좌측 패널·툴바·인스펙터)에는 적합하지만, **사용자가 만드는 캔버스 자체**에는 적합하지 않다.

## 4. React가 아닌 스택

ds는 React + TypeScript 전제로 설계됐다. Vue·Svelte·Solid·Angular·SSR-only(HTML) 환경에서 ds의 어휘를 이식할 수는 있지만, 정본·lint·audit 도구체인 전체를 옮기는 비용은 새로 만드는 것과 비슷하다. 그 스택에서는 그 스택의 de facto 표준(예: Vue용 Headless UI·Reka UI)을 따르는 편이 낫다.

## 5. LLM codegen을 도구로 쓸 의사가 없는 팀

ds의 모든 trade-off는 "LLM이 같은 결과로 수렴"이라는 결과를 위한 것이다. 팀이 LLM을 코드 생성·리뷰·migration·codegen 도구로 쓰지 않는다면 ds가 포기한 자유도(variant·className·children 자유 조립)는 단순한 손실이다. 그 경우 Radix + Tailwind·shadcn·MUI 같은 사람-우선 시스템이 같은 작업을 더 적은 제약으로 처리한다.

## 6. 디자인 시스템 자체를 처음부터 만드는 팀

ds는 정본이 명확한 도착점이지, "디자인 시스템이란 무엇인가"를 학습할 자료가 아니다. 첫 디자인 시스템을 만드는 팀이라면 Material 3·Polaris·Carbon 같은 큰 시스템의 토큰·문서 구조를 먼저 읽고, 그 다음 ds의 [Why](../02-why/README.md) 시리즈를 비교 자료로 보는 편이 낫다.

## 7. 짧게 — 수명이 6개월 미만인 프로토타입

해커톤·POC·실험 라우트처럼 **수명이 짧고 폐기될 코드**라면 ds의 정본 정의 비용이 ROI를 못 맞춘다. 이 경우 ds 안에서도 시연/카탈로그 라우트 본문에는 raw role을 허용하는 예외(C5 임시)가 있지만, 그 예외 자체가 정본 외부에서 빨리 끝낼 코드를 위한 것이다. 폐기될 코드라면 처음부터 ds 외부에서 작성하는 것이 깔끔하다.

---

위 7개 중 하나에 해당하면 ds를 권하지 않는다. 그렇지 않다면 [Getting Started](../04-getting-started/README.md)로.
