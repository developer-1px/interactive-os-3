# 정본 갱신 절차

정본은 고정 결정이 아니라 운동이다. 도메인이 흔들리면 정본도 흔들린다. 다만 흔들린 뒤 다시 좁아진다 — 항상 더 좁은 쪽으로.

## 4단계

1. **반례 발견** — 정본으로 표현 부족한 사례가 코드에 등장한다. Phase 1 회귀 트리거.
2. **후보 평가** — 5개 기준으로 본다: de facto · 직렬화 · 선언성 · 단순성 · 확장.
3. **합의** — 매뉴얼 1행 갱신 + 변경 일자 기록.
4. **일탈 감사 재실행** — 새 정본 기준으로 코드 전체를 다시 본다. 수렴되거나, 임시·유산으로 분류된다.

## 일탈 감사 명령

```
/canonical <영역명>     # 해당 영역 일탈만 감사
/canonical              # 매뉴얼 전체 항목 순회
```

`/canonical` 스킬이 Phase 0에서 [`CANONICAL.md`](../../../CANONICAL.md)를 자동 로드한다.

## 누가 결정하는가

지금 단계의 ds는 사실상 **1인 합의 + 반례 압력**의 조합이다. 솔직히 인정할 부분.

사람 1명이 결정해도 자의적이지 않은 이유는 **절차가 닫혀 있기 때문**이다.

- (a) **반례 트리거** — 코드에 실제로 일탈이 등장해야 회귀가 발동한다. 머릿속 가설로는 정본을 못 흔든다.
- (b) **고정된 후보 평가 기준** — de facto · 직렬화 · 선언성 · 단순성 · 확장 5개. 더 늘리거나 줄이지 않는다.
- (c) **결정의 박제** — 결정은 [`CANONICAL.md`](../../../CANONICAL.md) 1행으로 박히고 누구나 git diff로 추적 가능하다.

즉 결정자가 1명이어도 **결정 행위 자체는 직렬화**된다. 이것이 "권력 집중"과 "절차 직렬화"의 차이다.

팀이 늘어나면 자연스럽게 RFC PR + 1주 코멘트 + 합의 머지 형태로 옮겨갈 자리가 비어 있다. 지금은 비어 있다는 사실 자체를 인정하는 게 정본에 부합한다.

## 외부 contributor 정책 (가설)

ds는 아직 외부에 본격 공개되지 않았다. 외부 PR 정책은 가설 단계지만 3트랙으로 갈리도록 설계되어 있다.

1. **정본 안 PR** — lint 통과 + 정본 행 안 바꿈. 일반 리뷰. 빠르게 머지.
2. **정본 일탈 PR** — lint 실패. 자동 거부. "정본 X행 위반 — 수정하거나 정본 갱신 제안 PR로 분리하라" 봇 코멘트.
3. **정본 갱신 제안 PR** — `CANONICAL.md` 행을 직접 수정. RFC 트랙. 위 4단계 통과 필요.

이 3트랙이 있으면 외부 contributor가 정본을 "흔드는" 행위가 정상 흐름의 한 종류로 흡수된다. 흔드는 게 문제가 아니라, **흔드는 형식이 갖춰졌느냐**가 문제다.

정본은 거부의 *사유*가 아니라 거부의 *형식*이다. PR을 거부할 때 "내가 싫어서"가 아니라 "C2 직렬화 위반"이라고 말할 수 있어야 하고, 그 위반은 자동 lint로 객관화된다 — [04 · Lint 1:1 매핑](./04-lint-rules.md).

동의하지 않는 contributor는 (a) 정본 갱신 PR을 시도하거나 (b) fork한다. 두 길 다 정본의 권위와 자율성을 동시에 지킨다.

## 임시 / 유산 전이

정본 행은 종종 **임시**·**유산** 분류를 함께 가진다.

- **임시** — 지금은 허용. 만료 조건이 충족되면 정본으로 흡수되거나 유산으로 떨어진다. 예: "JSX Row/Column/Grid 조립 (이유 명기 시)".
- **유산** — 마이그레이션 대상. 새로 만들지 않고, 발견하면 정본으로 끌어당긴다. 예: "직접 flexbox 조립", "matchMedia 훅".

임시 → 정본 / 임시 → 유산 / 유산 → 제거 전이는 위 4단계를 그대로 따른다. 만료 조건은 임시 행에 함께 적힌다.
