# 3.7 Code Ownership Model — 같은 모델, 다른 함의

shadcn처럼 사용자가 컴포넌트 코드를 소유하는 모델이 ds와 충돌하나? 정답은: **소유 모델 자체는 충돌하지 않는다**. ds도 사용자 레포에 코드가 들어 있다(`packages/ds/src/ui/`). 충돌하는 건 "소유했으니 자유롭게 변형하라"는 **소유의 함의**다.

## shadcn의 소유 함의

- 컴포넌트 소스는 사용자 코드다 → 사용자가 패치·variant 추가·className 확장을 자유롭게 한다.
- "프레임워크의 의견을 빌려 시작하되 결국 너의 시스템으로 진화시켜라."
- 그래서 cva variant·Tailwind class를 코드 안에 직접 만지는 게 권장 사용법이다.

## ds의 소유 함의

- 컴포넌트 소스는 사용자 레포에 있지만 **정본**이다. `packages/ds`는 워크스페이스 패키지로 분리돼 있다.
- 변형은 정본 갱신 절차로만 가능하다 — 1인 합의·Phase 1 회귀·매뉴얼 1행 갱신.
- 변형이 정본화되지 않으면 임시 또는 유산으로 분류돼 추적된다.

두 모델 모두 "벤더 락인 거부"라는 점에서는 같다. 하지만 락인을 거부하는 **이유**가 다르다. shadcn은 "사용자의 자유를 위해", ds는 "정본을 사용자가 audit·patch·통제하기 위해". 결과적으로 ds 사용자에게 코드 소유는 권한이라기보다 **책임**이다.

## 무엇이 충돌하는가

shadcn 사용자가 자연스럽게 하는 행동 중 ds 정본을 위반하는 것들:

- "내가 owner니까 cva variant 추가할게" → ds P1 위반(1 role = 1 component, variant 금지)
- "Tailwind class로 살짝 다르게" → ds P2 위반(classless)
- "이 컴포넌트는 내 프로젝트 specific이니까 따로 둘게" → ds 정본의 "사용처 1곳도 entity 승격, 정본은 공용" 원칙과 마찰

## 무엇이 충돌하지 않는가

- "코드가 사용자 레포에 있다"는 모델 자체
- "벤더가 강제 업그레이드하지 않는다"
- "코드를 audit·debug할 수 있다"
- monorepo workspace 구조

## 같은 컴포넌트를 패치한다면

**shadcn 권장 흐름**:

```tsx
// components/ui/button.tsx — 내 코드니까 자유롭게 수정
const buttonVariants = cva('...', {
  variants: {
    intent: {
      primary: '...',
      destructive: '...',
      ghost: '...',
      brand: 'bg-purple-700 ...', // 새 variant 추가
    },
  },
});
```

**ds 정본 갱신 절차**:

1. 반례 발견 (정본으로 표현 부족)
2. 후보 평가 (de facto·직렬화·선언성·단순성·확장)
3. 사용자 합의 → 매뉴얼 1행 갱신 + 변경 일자 기록
4. 일탈 감사 재실행 → 수렴 또는 임시/유산 분류

shadcn은 코드 변경이 곧 변경이다. ds는 코드 변경 + 매뉴얼 1행 + 일자 기록이 함께 가야 정본이 된다. 그렇지 않으면 lint가 잡거나 임시/유산으로 분류된다.

## 정직한 한계

- 정본 갱신 권한자가 누구인지에 따라 "소유 = 책임"의 무게가 달라진다. 1인 합의 모델이면 다른 사용자에게는 사실상 락인이다.
- 오픈소스로 ds를 풀었을 때 외부 contributor의 PR이 정본을 흔드는 문제는 같은 축의 미해결 질문이다.

## 어떤 팀에 ds 모델이 맞나

- "코드 소유 = 책임"을 받아들일 수 있는 팀
- 정본 갱신 절차를 1인이라도 운용할 의지가 있는 팀
- 변형이 일어나면 매뉴얼·lint·날짜 기록까지 함께 가야 한다고 보는 팀

## 어떤 팀에 shadcn 모델이 맞나

- "코드 소유 = 자유"가 자연스러운 팀
- 컴포넌트 코드를 프로젝트별로 자유롭게 진화시키고 싶은 팀
- 정본 갱신 절차의 운용 부담을 받아들이고 싶지 않은 팀
- 컴포넌트가 프로젝트별로 갈라지는 게 자연스럽다고 보는 팀
