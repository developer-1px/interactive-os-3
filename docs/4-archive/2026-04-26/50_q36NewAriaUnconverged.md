# Q36 — 새 ARIA pattern, 라이브러리 수렴 없을 때

## 질문

ARIA APG에 새 pattern이 들어왔는데(예: 최근의 Carousel 개정, ComboBox 변경) 라이브러리 4곳 중 어디도 따라잡지 못했다. 채택? 보류?

## 해보니까 알게 된 것

- **기본은 보류.** 정본 명명 룰("최소 2곳 수렴")의 정신은 "한쪽이 한 번 시도한 형태로 도장 찍지 않는다"이다. ARIA 스펙 자체는 형태가 아니라 의미(role·aria-*) 정의라서 "스펙 채택"과 "컴포넌트 형태 채택"은 다른 사건.
- **의미는 즉시 채택, 형태는 보류**가 칼이다. 새 ARIA role/aria-*는 ds 컴포넌트가 그 role을 출력하는 데 즉시 쓴다(왜냐면 보조기기에 노출되는 진실이라서). 컴포넌트 이름·prop 시그니처는 라이브러리 2곳이 수렴할 때까지 ds 내부 임시 이름으로 둔다.
- **임시 표시는 명시적으로**. 임시는 CANONICAL.md "임시" 컬럼에 만료 조건과 함께 적는다 — "라이브러리 N곳 수렴 시 갱신". 임시인 채로 묻혀있는 걸 막는다.
- 사용처가 1곳이라도 entity처럼, 사용처가 1곳이라도 임시 컴포넌트로 만들 수 있다 (C6: 정본 ≠ 이상형이 막는 건 "안 쓰는 정본"이지 "쓰는 임시"가 아니다).
- 위급한 채택 압력(법적 접근성 요구·실 사용자 보고)이 있으면 보류 안 하고 임시로 즉시 채택. 정본은 후행.

## 근거

- CANONICAL.md "컴포넌트 명명" + "임시/유산" 컬럼.
- 헌장 C5 "이름이 곧 셀렉터" — role/aria는 의미 layer이므로 즉시 정확해야 함.
- MEMORY.md `feedback_prefer_de_facto` — 수렴 없는 단독 사례는 채택 안 함.

## 남은 의문

- "임시"가 결국 영구가 되는 사례(라이브러리들이 영영 안 따라옴) 처리. 1년·2년 임시면 ds가 사실상 정본 후보 발신자가 되는 셈인데, 그건 "발명하지 않는다" 원칙과 충돌.
- ARIA 스펙도 의견이 갈리는 working draft 단계 변경은? "안정화 전에는 안 본다" 정책이 필요할 듯.
- 임시 컴포넌트의 LLM 노출 — 임시도 코드에 있으면 LLM은 그걸 정본처럼 쓴다. "임시" 표시를 LLM이 읽도록 강제하는 메커니즘이 없다.
