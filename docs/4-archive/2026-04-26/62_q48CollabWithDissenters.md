# Q48 — ds 받아들이지 않은 사람과 협업?

## 질문

why 시리즈를 읽고 "동의 안 한다"는 사람과 같은 코드베이스에서 일해야 할 때 어떻게? 정본이 권력으로 작동하면 협업이 깨진다. 권력으로 작동 안 하면 정본이 무너진다.

## 해보니까 알게 된 것

- 동의 ≠ 준수. ds는 동의를 요구하지 않는다. **lint 통과를 요구한다**. 사상에 동의 안 해도 정본 lint는 통과시킬 수 있다.
- 이게 ds의 사회적 트릭: 정본을 lint로 객체화함으로써 사상 토론을 코드 머지 게이트와 분리한다.
- 협업 시 3가지 길:
  1. **동의 안 하지만 준수**: 가장 흔한 케이스. 정본 lint만 통과하면 됨. 사상은 본인 머릿속.
  2. **정본 갱신 제안**: 동의 안 하는 부분을 갱신 PR로 — Q45의 RFC 트랙. 반례 + 후보 평가로 정본을 흔들 정당한 길.
  3. **분리**: 코드 영역을 나눔. ds 정본을 따르는 영역(@p/ds·@p/app·core widgets) vs 자유 영역(특정 도메인 라우트 안). CANONICAL.md "임시" 표기와 시연 라우트 raw role 예외가 이미 그 모델.
- 무엇을 하면 안 되나:
  - "동의 안 하면 PR 거부" — 정본을 권력으로 만든다
  - "사상 합의 없이 머지 거부" — 정본을 신앙으로 만든다
  - 둘 다 lint를 우회하는 행위. 정본의 객체성이 무너진다.
- 솔직: 1인 프로젝트 단계에서는 이 질문이 가설이다. 다인 협업 들어오면 첫 충돌 사례가 정책의 입력이 된다.

## 근거

- CANONICAL.md L24 "C6 정본 ≠ 이상형" — 코드에 안 쓰이는 형태는 정본 못 박지 않음. 사상보다 코드 사용 우선.
- 메모리 [No escape hatches] vs [시연 라우트 raw role 예외] — 영역 분리가 이미 정본화됨
- Q45 외부 contributor 3트랙 — 동의 안 하는 자의 통로가 RFC 트랙으로 열림
- Q43 자동 lint 60-90% — "동의보다 통과" 모델의 기술적 기반

## 남은 의문

- 사상 토론용 별도 공간(Discussions·forum)을 머지 게이트와 분리해 마련할지
- 동의 안 하는 사람이 정본 lint를 우회하는 escape hatch를 발견했을 때 — 그 자체가 새 정본 입력
- 협업 onboarding 문서: "동의 강요 안 함, lint 통과만 강요"를 명시할지
- 1인 합의 구조에서 의견 충돌의 첫 사례를 어떻게 기록·박제할지 (decision log)
